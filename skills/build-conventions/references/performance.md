## 적용 범위

모든 스프린트 — 단, 성능 예산은 스프린트 특성(백엔드 API / 프론트 페이지 / 배치 등)에 따라 달라진다.

## 채우기 가이드

"빠르게 해라" 같은 문장 금지. **수치 예산(budget)**과 **측정 방법**을 함께 명시한다. 예산 없이 규칙을 걸면 판정 불가.

### 카테고리 체크리스트
- [ ] **번들 크기** (프론트): initial JS/CSS 상한
- [ ] **렌더 성능** (프론트): LCP, CLS, INP 목표
- [ ] **API 응답** (백엔드): p95 응답 시간
- [ ] **메모리**: 장기 실행 누수 방지, 최대 heap
- [ ] **쿼리 효율**: N+1 금지, 인덱스 전제
- [ ] **빌드 시간**: 증분 빌드 목표 (개발자 경험)

## 측정 도구 (참고)

- 프론트: Lighthouse, Web Vitals, webpack-bundle-analyzer
- 백엔드: 부하 테스트 도구(k6, wrk), APM
- 공통: 명시적 assertion이 있는 테스트 (응답 시간 측정)

### (예시) R1. 초기 JS 번들 ≤ 200KB (gzip)
- **선언:** 첫 화면 렌더에 필요한 JS의 gzip 크기가 200KB를 넘지 않는다.
- **검증:** 빌드 후 번들 분석 도구로 initial chunk 크기 측정. 값 기록.
- **예외:** Feature-spec에 "대형 클라이언트 라이브러리 불가피" 명시 시 재협상
- **위반 시 판정:** WARN (+20% 이내), FAIL (+20% 초과)

### (예시) R2. API p95 응답 ≤ 300ms
- **선언:** 대표 엔드포인트 중 하나라도 p95 > 300ms면 스프린트 FAIL.
- **검증:** 부하 테스트 스크립트 실행, 결과 로그 첨부
- **예외:** 배치·비동기 엔드포인트는 별도 규칙 적용
- **위반 시 판정:** FAIL
