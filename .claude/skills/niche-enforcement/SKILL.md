# Niche Enforcement — 3D 게임 범위 확장 차단 프로토콜

## 핵심 규칙: 1플레이어 원형 · 1코어 결핍 · 1코어 버브

| 차원 | 정의 | 거부 신호 |
|------|------|----------|
| **1플레이어 원형** | 이 게임을 실제로 90분 쓸 구체 인물 1명. 연령·플레이 세션 길이·플랫폼·게임 경험 중 최소 3개 명시. | "코어 게이머와 캐주얼 모두", "모든 연령층", 플랫폼 복수(PC+모바일+콘솔 동시), "누구나" |
| **1코어 결핍** | 이 원형이 현 시장에서 못 채우는 **감정적 결핍 1개**. 긴장 해소 / 탐험 충동 / 성취감 / 사회적 연결 / 표현 욕구 중 하나. | "스트레스도 풀리고 몰입도 되고", "재미+학습+소셜", 복수 감정 나열 |
| **1코어 버브** | 플레이어가 60분 세션에서 100번+ 수행하는 동사 1개. (쏜다/탐험한다/짓는다/피한다/수집한다/대화한다/베어낸다 등) | "탐험하면서 전투하고 건설", 장르 짬뽕, 세컨더리 버브를 코어로 승격, "그리고" 연결 |

**판정 시 `game-design-loop` 스킬을 함께 로드한다.** 거기에 각 차원의 구체 검증 질문(코어루프 4단계 정합성·세션 곡선·학습 곡선)이 있다.

## 판정 절차 — 순차 확정 원칙

**세 차원을 한 번에 판정하지 않는다.** Phase 1에서는 **1플레이어 원형 → 1코어 결핍 → 1코어 버브** 순서로 각 차원이 독립 판정되며, 앞 차원이 확정되기 전에 다음 차원을 판정 대상으로 받지 않는다. 3요소가 섞인 제안이 들어오면 "먼저 1플레이어 원형부터 확정합시다"로 되돌린다.

Phase A(토론 중심)에서는 각 토픽 사이클의 **5단계 수렴 스텝**에서 Niche Enforcer가 veto 판정을 내린다. 토론 중 다른 에이전트가 범위를 확장하는 발화를 하면 즉시 제지.

### Step 1: 대상 문서/제안 읽기
- why.md, what.md, how.md, Feature 제안 등 판정 대상을 전부 읽는다.
- 필요 시 이전 `niche_verdicts.md`를 참조해 이미 판정한 항목을 재확인한다.
- Phase 1 단독 차원 판정이면 해당 차원(`confirmed_player.md` 등) 하나만 본다.

### Step 2: 차원별 판정 (한 번에 하나씩)

각 차원은 **독립 라운드**로 판정한다. 한 차원이 APPROVED 되기 전까지 다음 차원으로 넘어가지 않는다.

#### 1플레이어 원형 판정 질문
- "이 게임을 여는 구체 인물 1명"을 한 문단으로 그릴 수 있는가? 연령·세션 길이·플랫폼·게임 경험 중 **최소 3개** 명시됐는가?
- 페르소나에 "코어 게이머와 캐주얼 모두" 같은 범위 확장이 있는가? → 거부.
- 플랫폼이 2개 이상(예: PC+모바일) 동시 선언됐는가? → 첫 플랫폼 1개만 선택하도록 재협상.
- "모든 플레이 스타일 수용", "누구나 즐길 수 있는" 표현이 있는가? → 거부.

#### 1코어 결핍 판정 질문
- 이 원형이 현 시장에서 **채우지 못하는 감정**을 한 문장으로 답할 수 있는가?
- 감정이 여러 개(재미+학습+소셜 등)로 나열되는가? → 거부, 가장 지배적인 것 하나만.
- 감정의 "채워지지 않는 이유"가 시장 증거(Steam 리뷰 반복 불만·YouTube 댓글·커뮤니티 포스트)로 뒷받침되는가?
- 결핍이 측정 가능한가? (예: "긴장 해소"라면 유사 게임의 리텐션/세션 길이로 간접 측정 가능)

#### 1코어 버브 판정 질문
- 플레이어가 가장 자주 하는 **동사 1개**를 지목할 수 있는가?
- 60분 세션 동안 **100번+ 수행** 가능한가? (아니면 코어가 아닌 세컨더리)
- **Time-to-First-Verb 3분 이내**로 경험 가능한가? (아니면 진입장벽)
- 2개 이상의 버브가 "코어"로 선언되었는가? → 즉시 거부.
- 코어 버브가 코어루프 4단계(Anticipation → Action → Feedback → Progress)에 꽂히는가? (Action 단계에 명확히 위치해야 함)

### Step 3: 판정 기록

`docs/kickoff/_workspace/niche_verdicts.md`에 다음 형식으로 기록한다:

```markdown
# Niche Enforcement Verdicts

## Round {N} — YYYY-MM-DD

### Target: {document or proposal name}

| 차원 | 판정 | 근거 | 축소 제안 |
|------|-----|-----|----------|
| 1플레이어 원형 | ✓ / ✗ | ... | (거부 시) ... |
| 1코어 결핍 | ✓ / ✗ | ... | (거부 시) ... |
| 1코어 버브 | ✓ / ✗ | ... | (거부 시) ... |

**결정:** APPROVED / CHANGES_REQUESTED / REJECTED
**재협상 시 고려:** ...
```

## 판정 포맷 3단계

- **APPROVED**: 세 차원 모두 통과. 다음 토픽/문서로 진행 가능.
- **CHANGES_REQUESTED**: 일부 차원에서 구체 수정 필요(문구 조정·1개 요소 삭제). 수정 후 재판정.
- **REJECTED**: 본질적 범위 확장. 해당 토픽을 다시 토론해야 함(Phase A 사이클 재실행).

## 거부 시 원칙

1. **추가가 아닌 삭제로 유도한다.** "X를 더 넣자"가 아니라 "Y를 빼면 통과"로 제안.
2. **대안 축소안 2~3개를 제시한다.** 팀이 선택하게 한다.
3. **Founder도 뒤집지 못한다.** Niche Enforcer의 거부는 Kickoff 팀 내 최종 결정 (veto 권한).
4. **경계 사례는 거부가 기본.** 애매하면 빼라.

## 자동 거부 패턴 (즉시 REJECTED)

아래 패턴은 논의 없이 즉시 REJECTED 처리한다.

| 패턴 | 거부 이유 | 축소안 |
|------|----------|-------|
| **코어 버브 2개+** ("탐험하면서 전투하고 수집") | 1코어 버브 위반 | 60분/100번+ 테스트로 실제 코어 버브 1개 식별, 나머지 세컨더리로 |
| **플랫폼 2개+ 동시** (PC+모바일+콘솔) | 1플레이어 원형 위반, 입력 모델 다름 | 첫 플랫폼 1개. 나머지는 post-launch 순차 이식 |
| **장르 3중 혼합** ("로그라이크 + 빌딩 + 서바이벌") | 학습 곡선 파괴, 코어루프 분산 | 주 루프 1개로 수렴, 나머지는 세컨더리 또는 삭제 |
| **솔로 + 멀티 동시** (MVP에서) | 설계 표면 2배, 1플레이어 원형 위반 | MVP는 솔로만. 멀티는 post-launch |
| **코어루프 무관 Feature** (스토리·UI 개선이 Anticipation/Action/Feedback/Progress 어디에도 안 꽂힘) | 코어 버브를 지탱하지 않음 | Feature 삭제 또는 코어루프 기여 섹션 재작성 |
| **레퍼런스 게임 체리피킹** ("엘든링의 전투 + 젤다의 탐험 + 스타듀의 농사") | 복수 코어 버브 은폐, 현실적 구현 불가 | 하나의 레퍼런스를 1차로 택하고 나머지는 "가벼운 양념"으로만 |
| **"모든 플레이 스타일 수용"** | 원형 범위 확장 | 하나의 원형(공격형 or 수비형)에 최적화, 나머지는 viable하되 optimal 아님 |
| **"언리얼/유니티 이식 가능하게"** | 엔진 추상화는 범위 외 | Godot 4 전용. 이식은 post-launch |
| **"나중에 콘솔 이식 예정"** (how에 이미 가정) | 범위 외부를 미리 약속 | 로드맵에만 기술. why/what/how에서 제거 |

## 자주 보이는 CHANGES_REQUESTED 패턴

| 패턴 | 수정 요구 |
|------|----------|
| "FPS이자 RPG 요소도" | 주 버브 1개(쏘다 or 성장시키다) 선택 명시, 반대편은 "가벼운 양념"으로 재기술 |
| "감동적이고 긴장감 있고 몰입감 있는" | 가장 지배적 감정 1개 명시, 다른 감정은 그 감정을 뒷받침할 때만 등장 |
| 플레이어 원형이 "30대 게이머" 수준 | 연령·세션 길이·플랫폼·게임 경험 중 3개 이상 구체화 |
| North Star 지표가 정성적 ("재밌는 게임") | 숫자 목표 (Retention %·Review Score·동시접속자 등) 명시 |
| Feature가 코어루프 기여 명시 안 됨 | Anticipation/Action/Feedback/Progress 중 어느 단계 지탱하는지 표기 추가 |

## 승인 후에도 감시

승인 이후 Kickoff 팀이 how 작성 중 범위 확장을 시도할 수 있다. Niche Enforcer는 how.md 초안에서도 동일 규칙으로 재판정한다. 이전 단계에서 승인한 범위도 how에서 은근히 확장되면(예: "Forward+ + Mobile 둘 다 지원", "Autoload 7개로 확장", "멀티 대비 네트워크 레이어 미리") 거부한다.

Feature Phase(feature-spec)에서도 동일. 개별 Feature가 what.md에 없던 신규 코어 버브를 은근히 도입하면 거부.
